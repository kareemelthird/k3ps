## Phase 8 — Offline-first hardening (the durable write queue) — ACTIVE PHASE, specced → architect → build
> Spec: [`docs/specs/phase-8-offline-first-hardening.md`](specs/phase-8-offline-first-hardening.md) · Anchors: [CLAUDE.md §2.8](../CLAUDE.md) (idempotent writes — client UUIDs + upsert; queue survives crashes without double-counting), §2.1 (integer piastres), §2.2 (timers from timestamps), §2.4 (`@ps/core` pure), §5 (RLS/tenant isolation), §2.6 (Arabic-first RTL). Reference: [`docs/reference/mobile-patterns.md`](reference/mobile-patterns.md) (offline-outbox) + `offline-outbox-guard` skill. New decisions land as **ADR-0009** (offline outbox + realtime sync for multi-tenant).
> Key fact: **mobile + core only** (backend only if §7 elects a close RPC or realtime publication/RLS). Phases 3–7 mobile does **direct idempotent writes** (client `uuidv4`/`uuidv5` keys + upsert) but **not durable** — a network drop or app-kill mid-write loses bills / leaves the 5-step close sequence partial (Phase-5 "not atomic (durability)" residual). Phase 8 ports the trial's proven outbox (`D:\K3\Pochinki\src\features\sync\`) — durable queue, dead-letter, network watcher, realtime — rebuilt for tenancy, **never imported**. Lift the pure decisions (state machine / idempotency / ordering / retry / dead-letter) into `@ps/core` (>90%); keep I/O in mobile adapters. No new money math, no new business tables.
- 🟡 `@ps/core` `outbox/` (pure, >90%): `OutboxEntry` type (+`tenant_id`/`branch_id`/`dependsOn`); idempotent enqueue/dedupe; drain-selection with **dependency ordering**; retry/backoff policy + permanent-vs-transient error taxonomy; dead-letter decision; pure transition fns (`onSuccess`/`onTransientFailure`/`onPermanentFailure`). No React/RN/Supabase/storage imports; no `Date.now()` in decisions (purity test).
- 🟡 Mobile (Expo): durable crash-safe persistence (mechanism = §7 Q1: SQLite/MMKV/AsyncStorage) surviving force-kill + restart; `persistRow`/outbox bridge; **reroute every existing mutation** (session start/switch/close; order add/pay/void; stock restock/adjust/sale; shift open/close; device status) through the queue preserving payloads + `uuidv4`/`uuidv5` keys; ordered drain (parent before dependents); network watcher (`navigator.onLine`/AppState/web events — **no** cross-origin probe) + auto-drain on reconnect; tenant-scoped realtime invalidation; `useSync` store + rehydrate counts on launch; sync-status UI (pending/syncing/failed/last-synced + manual retry/discard, confirm on money discard).
- 🟡 Fresh RTL/Arabic-first sync UX (`ui-ux-pro-max` + magic MCP): glanceable sync indicator (online/offline + pending/failed badges + syncing + last-synced), sync detail screen (pending list, dead-letter list w/ errors, retry/discard), all-clear state for shift-close confidence, dead-letter alarm state; Arabic-Indic counts/times; extend existing `OfflineBanner`. No hardcoded strings.
- 🟡 architect decisions blocking build (spec §7 → ADR-0009): **persistence mechanism + crash-atomic write strategy** (SQLite vs MMKV vs AsyncStorage — cite Expo docs); pure-core/adapter boundary; **ordering/dependency model** (explicit `dependsOn` vs transactional close RPC vs FIFO — RPC fixes the non-atomic close but adds backend); retry/backoff + dead-letter thresholds + error taxonomy; **realtime tenant-scoping/authorization** (postgres_changes filter + Realtime Authorization RLS — client filter alone insufficient; cite Supabase docs — security blocker if it leaks); conflict-resolution per entity (LWW vs terminal/append-only); whether any backend change is needed (target: none beyond optional realtime publication / close RPC).
- ⚪ Out of scope (later phases): Stripe billing/plans/paywall (9); Sentry/EAS/perf/a11y/pen-test (10); web offline support (web stays online-only; realtime auto-refresh only if trivially shared); full offline read cache / offline history browsing; cross-device manual merge UI; OS background-terminated sync (drains on next launch/foreground/reconnect); new operational features / new money math / new business tables; multi-currency/timezone generalization.
